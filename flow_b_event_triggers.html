Architecture Change

Flow B —
Event-Driven Triggers

Flow B now fires automatically when qualifying data changes — no manual client call required. COMPASS scores stay current with each wearable sync and every clinical data update.

📅 Date: 2026-04-29
🏭 Platform: NUR Backend + NURAI
📄 Ref: system_design.html §7.5
Section 01

Overview

Flow B is the COMPASS wearable refresh pipeline — it recomputes an athlete's readiness score on days without a new thermal scan, using the latest wearable metrics and clinical context. Previously, Flow B only ran when a client explicitly called POST /compass/refresh. This document describes the change that makes Flow B event-driven: it now fires automatically from data-write endpoints whenever qualifying data changes.

⚡

Trigger 1 — Wearable Data

The daily Spike pull job and the on-demand pull endpoint now trigger Flow B automatically for each user whose daily_metrics row was updated with new data.

💊

Trigger 2 — Health & RTP Data

Pain entries, RTP workflow writes (initiate, injury, intervention, clearance), and health profile updates all trigger Flow B automatically for the affected athlete.

🔒

Gate Logic — Unchanged & Centralised

All Flow B gate checks (scan validity, wearable freshness, cached status) remain inside NURAI — identical to how POST /compass/refresh works today. No gate logic runs at the trigger site.

⚙

No New Infrastructure

No Cloud Tasks queue, no new GCP services, no new database tables. Each trigger spawns a background thread that calls the existing NURAI POST /infer/refresh endpoint directly.

Section 02

Before & After

The core change is moving from a pull model (client asks for a refresh) to a push model (the backend triggers a refresh when data changes).

Before — Manual Pull
  • COMPASS score only updated when My NUR or NUR HUB explicitly calls POST /compass/refresh
  • Wearable data arrives via daily Spike pull — score stays stale until client calls refresh
  • Pain entry or RTP update has no effect on the current COMPASS score until next manual refresh
  • Client must know when to call refresh to keep data current
  • Score displayed to athlete may be hours or days behind the latest wearable data
After — Event-Driven
  • COMPASS score updated automatically when new wearable data arrives from Spike pull
  • Score updated automatically when athlete or staff submits a pain entry
  • Score updated automatically when any RTP workflow action is taken
  • Score updated automatically when athlete updates health profile
  • On app open, GET /results/latest always returns the most recently computed score

Key insight: The triggering endpoint or job returns its response to the caller immediately — no blocking on NURAI. The background thread handles the NURAI call asynchronously. POST /compass/refresh is preserved for manual refresh and serves as the natural recovery path if a background trigger fails.

Section 04

Background Execution Pattern

All triggering endpoints follow the same pattern. The endpoint's own work completes first and returns immediately to the caller. NURAI runs in a background thread — the caller never waits for the COMPASS computation.

  • 1

    Primary DB write. The endpoint completes its own responsibility: upsert daily_metrics (wearable), insert new user_profile_history version (pain/profile), or write to rtp_workflows / rtp_interventions (RTP).

  • 2

    Background thread spawned. The endpoint spawns a background thread carrying user_id and trigger_type. The endpoint then returns its normal response to the caller — the caller is not blocked.

  • 3

    NURAI called. The background thread calls NURAI POST /infer/refresh with { user_id, trigger_type: "auto_wearable" | "auto_health" }. NURAI runs the full Flow B gate sequence (§7.2).

  • 4

    Module 2 runs only if gates pass. If all gates pass, NURAI runs the Metrics Pipeline (Module 2) to recompute zone and score from the latest wearable data. If any gate fails, Module 2 is skipped — the existing COMPASS score and zone are unchanged. Either way, Module 3 always runs (see step 5).

  • 5

    Module 3 always runs. GenAI Reasoning executes regardless of gate outcome. When gates passed, Module 3 runs with the freshly computed score and zone from Module 2; NURAI writes the updated score, zone, and ai_coach_response to daily_assessments, and the Alert Generator fires a zone_change alert if the zone shifted. When gates failed, Module 3 runs using the last valid score and zone; only ai_coach_response is updated — score and zone remain as-is.

  • 6

    Client picks up result. On the client's next GET /results/latest (app open or manual refresh), the updated daily_assessments record is returned. No polling needed — the computation is typically complete within seconds.

Failure Behaviour

⚠
Background thread failure

If the NURAI background thread fails or times out, the triggering endpoint's response is unaffected — it already returned to the caller. The daily_assessments row is unchanged; the client sees the last known valid score. The client's next POST /compass/refresh or app-open GET /results/latest serves as the natural recovery path. No retry mechanism or dead-letter queue is needed.

Section 05

Gate Logic — Unchanged, Centralised in NURAI

No gate checks run at the trigger site. The triggering endpoint simply spawns a background thread and calls NURAI. All gate logic lives inside the Flow B pipeline in NURAI — identical to how POST /compass/refresh works today.

GateCheckFail resultRuns for
Gate 1 No thermal scan on record for this user no_scan — Module 2 skipped, Module 3 runs All trigger types
Gate 2 Latest valid scan is older than 7 days stale_scan — Module 2 skipped, Module 3 runs All trigger types
Gate 3 daily_metrics is not newer than the last computation cached — Module 2 skipped, Module 3 runs auto_wearable and manual wearable_refresh only — bypassed for auto_health
ℹ
Trigger site does not inspect refresh_status

The background thread fires the NURAI call and exits. NURAI handles all gate outcomes: gate pass → Module 2 + Module 3 run; gate fail → Module 2 skipped, Module 3 still runs. There is no callback to the triggering endpoint.

Section 06

Gate 3 Bypass for auto_health

Gate 3 exists to prevent redundant score computation when wearable data hasn't changed. It checks whether daily_metrics is newer than the last computation — if not, it returns cached and skips Module 2 (though Module 3 still runs).

A pain entry, RTP write, or health profile update does not change daily_metrics. Without a bypass, Gate 3 would always block Module 2 for health-triggered calls — the COMPASS score would never be recomputed to reflect the new clinical context (only ai_coach_response would update). The bypass ensures the score itself also incorporates the updated health state.

✓
Resolution: Gate 3 is bypassed when trigger_type = "auto_health"

NURAI skips Gate 3 for health-triggered calls and proceeds directly to Module 2 if Gates 1 and 2 pass. This ensures new clinical context (updated pain level, injury documentation, RTP state, activity level) is reflected in the recomputed COMPASS score — not just in the GenAI narrative. Without the bypass, only ai_coach_response would update; the score and zone would remain stale.

Section 07

trigger_type Enum — Schema Change

Two new values are added to the trigger_type column on tbscan.daily_assessments. Existing values are unchanged.

ValueStatusSet byDescription
new_thermal_scan Existing NURAI after Flow A (athlete QR scan) Full pipeline run following a new athlete-initiated thermogram.
admin_scan Existing NURAI after Flow A (admin scan) Full pipeline run following an operator-initiated thermogram via NUR HUB.
wearable_refresh Existing NURAI after manual POST /compass/refresh Flow B triggered manually by My NUR or NUR HUB.
auto_wearable New NURAI after wearable_updated background trigger Flow B triggered automatically by the daily Spike pull job or on-demand pull after new daily_metrics data is written.
auto_health New NURAI after health_updated background trigger Flow B triggered automatically after a pain entry, health profile update, or RTP workflow action.

The trigger_type value is forwarded by the Backend's background thread to NURAI in the POST /infer/refresh request body and written by NURAI into daily_assessments.trigger_type on upsert.

Section 08

Wearable Trigger Endpoints

Both the scheduled and on-demand Spike pull paths now spawn a background Flow B trigger after successfully writing new wearable data for a user.

POST
/internal/cron/spike-pull
Daily Spike pull job. Runs once per 24 hours. Iterates all queued users (those with a record_change webhook since the last pull), fetches Spike Daily Statistics API for each, and upserts daily_metrics.
⚡ Flow B trigger: spawned per user after Step 4 (daily_metrics written). Users with no new data skipped. Each thread independent — job does not wait.
POST
/wearables/sync-now
On-demand Spike pull. Athlete or coach initiates an immediate pull outside the daily schedule. Rate-limited to one successful pull per athlete per hour. Returns { last_synced_at, new_data_available, ready_for_compass_refresh }.
⚡ Flow B trigger: spawned in Step 4 on success when new_data_available = true. If background trigger completes before client manually calls POST /compass/refresh, NURAI Gate 3 returns cached — no duplicate computation.

Section 10

Client Behaviour

Clients do not need to know when a background trigger fired. They call GET /results/latest on app open or manual refresh and always receive the most recently computed score. No polling loop, no push notification, no special handling.

📷

My NUR — On App Open

GET /results/latest returns the most recent daily_assessments record. The trigger_type field tells the app how the score was computed — display context accordingly.

auto_wearable → "Updated after wearable sync"
auto_health → "Updated after health entry"
wearable_refresh → "Manually refreshed"
🔄

My NUR — Manual Refresh

POST /compass/refresh works exactly as today. If the background trigger already completed, NURAI Gate 3 returns cached immediately — no duplicate computation.

👥

NUR HUB — Readiness Dashboard

Existing GET /teams/readiness poll (every 60 seconds) picks up updated scores automatically. No behaviour change required.

📌

NUR HUB — After Pain Entry

Staff submits pain entry → endpoint returns 200 → show "Severe pain recorded. COMPASS will recalculate automatically." → next 60s poll reflects updated score.

Section 11

Key Design Decisions

DecisionChosen approachWhy
Async queue vs. inline background thread Inline background thread per trigger No new GCP services, no infrastructure to operate. NURAI calls are typically fast (<120s). Cloud Tasks / Pub/Sub would add operational complexity with marginal reliability benefit at this stage.
Gate logic location All gates inside NURAI, none at trigger site Gates already exist in NURAI for POST /compass/refresh. Duplicating them at each trigger site would create two code paths to keep in sync. Single location = single source of truth.
Gate 3 for health triggers Bypass Gate 3 when trigger_type = "auto_health" Gate 3 guards against recomputing on unchanged wearable data. Health writes change clinical context but not wearable data, so Gate 3 would always block health triggers — the bypass is required for the trigger to have any effect.
Push notification vs. pull Client pulls on app open / manual refresh Push infrastructure not yet in place. Pull is simpler, and since background triggers complete in seconds, the score is ready by the next natural client interaction.
New DB table for dedup/audit None — no trigger_log table NURAI's upsert on UNIQUE (user_id, date) in daily_assessments is the natural idempotency mechanism. A separate log table adds complexity with no clear operational payoff until volume demands it.
Retry on background thread failure No retry — POST /compass/refresh is recovery The triggering endpoint already returned success to the caller. A silent background failure is acceptable: the client gets the last known score, and the next manual refresh or app open recovers naturally.
