Schema Change · tbscan.thermal_scans

thermal_scans — Modified

The measurements JSONB column is extended to carry three new groups of per-region fields. Five new top-level columns are added.

Extended measurements JSONB — Per-Region Schema

Fields are split into two write phases. Module 1 writes synchronous fields immediately. The new async Baseline + RTP thread writes the remaining fields after scan delivery.

FieldWritten byDescription
temp_leftModule 1 (sync)Surface temperature of left side of region (°C).
temp_rightModule 1 (sync)Surface temperature of right side of region (°C).
deltaModule 1 (sync)Absolute bilateral temperature delta (°C).
affected_sideModule 1 (sync)Side with higher temperature: "left", "right", or "center".
severity_level newModule 1 (sync)Bilateral delta classification: clear (<0.3°C) · mild (0.3–0.5°C) · moderate (0.5–0.8°C) · elevated (≥0.8°C).
baseline_mean_delta newAsync threadRolling mean of per-region deltas across qualifying historical scans. null until thread completes.
baseline_sd newAsync threadRolling standard deviation of per-region deltas. Used in the individual SD baseline formula at ≥ 6 qualifying scans.
qualifying_scan_count newAsync threadNumber of historical scans used to compute the baseline. Drives N/A threshold (min 2) and formula switch (min 6).
rtp_clear newAsync threadRTP clearance status for this region. See logic below.

rtp_clear Logic

  • N/A

    Insufficient History

    qualifying_scan_count < 2 — not enough scans to evaluate the 2-consecutive-scan rule.

  • YES

    Non-Affected Side — Auto-Clears

    This region's anatomical side ≠ affected_side. The contralateral (healthy) side is automatically cleared regardless of delta history.

  • YES

    Affected Side — Mild / Clear for 2 Consecutive Scans

    Region side = affected_side AND severity_level was mild or clear (delta < 0.5°C) for the last 2 consecutive scans.

  • NO

    Affected Side — Still Elevated

    Region side = affected_side AND at least one of the last 2 scans has severity_level ≥ moderate (delta ≥ 0.5°C).

Full Example — Region 12 After Async Thread

{
  "12": {
    "temp_left":             36.2,
    "temp_right":            35.8,
    "delta":                 0.4,
    "affected_side":         "left",
    "severity_level":        "mild",
    "baseline_mean_delta":   0.31,
    "baseline_sd":           0.08,
    "qualifying_scan_count": 8,
    "rtp_clear":             "NO"
  }
}

New Top-Level Columns on thermal_scans

ColumnTypeDescription
max_asymmetry newnumeric(4,2), nullableMaximum absolute temperature delta across all detected regions in measurements. Written synchronously by Module 1. NULL if measurements is NULL. Enables fast threshold filtering without JSONB inspection.
has_asymmetry newboolean, nullabletrue if any region in measurements has affected_side not null. Written synchronously by Module 1. NULL if measurements is NULL.
rtp_cleared newboolean, nullableNULL until the async Baseline + RTP Thread completes. true if all per-region rtp_clear values in measurements are "YES". false if any region has rtp_clear = "NO". Written by the async thread alongside rtp_regions_pending.
rtp_regions_pending newsmallint, nullableNULL = async thread not yet complete. 0 = all regions clear. >0 = count of regions where rtp_clear = "NO". Enables fast RTP Hub queries without JSONB scanning.
baseline_computed_at newtimestamptz, nullableTimestamp when the async Baseline + RTP thread last updated this scan row. NULL = computation pending.

New Subsection · §6.6 Named Query Patterns

§6.6 Named Query Patterns — New

5 SQL query patterns used by NUR HUB API endpoints. No dedicated tables — all derived from existing schema. Added as a new subsection at the end of §6.

§6.6.1 — 30-Day COMPASS History

Used by GET /teams/athletes/{id}/compass/history. Reuses idx_ud_user_date (user_id, date DESC).

SELECT date, compass_score, compass_zone, trigger_type
FROM tbscan.daily_assessments
WHERE user_id = :athlete_id
ORDER BY date DESC LIMIT 30;

§6.6.2 — Contralateral Pair Deltas

Used by GET /teams/athletes/{id}/thermal/contralateral. Pairing: regions sharing a base label grouped as left/right pairs. Response shape: { pairs: [{ label, dates[], left_deltas[], right_deltas[], bilateral_diffs[] }] }. bilateral_diff = |left_delta − right_delta|.

SELECT ts.generated_at,
       ts.measurements->:left_region_id->>'delta'  AS left_delta,
       ts.measurements->:right_region_id->>'delta' AS right_delta
FROM tbscan.thermal_scans ts
WHERE ts.user_id = :athlete_id AND ts.is_valid = true
ORDER BY ts.generated_at DESC LIMIT 14;

§6.6.3 — Scan Overdue Detection

Used by the Alert Generator cron. Compares last valid scan date against org_settings.scan_interval_days.

SELECT om.user_id FROM tbscan.org_members om
LEFT JOIN (
  SELECT user_id, MAX(generated_at) AS last_scan
  FROM tbscan.thermal_scans WHERE is_valid = true GROUP BY user_id
) ts ON ts.user_id = om.user_id
JOIN tbscan.org_settings os ON os.org_id = om.org_id
WHERE om.org_id = :org_id
  AND (ts.last_scan IS NULL
       OR ts.last_scan < NOW() - INTERVAL '1 day' * os.scan_interval_days);

§6.6.4 — Wearable Gap Detection

Used by the Alert Generator cron. Flags active wearable connections not synced in over 36 hours.

SELECT wc.user_id FROM tbscan.wearable_connections wc
JOIN tbscan.org_members om ON om.user_id = wc.user_id
WHERE om.org_id = :org_id
  AND wc.status = 'active'
  AND wc.last_synced_at < NOW() - INTERVAL '36 hours';

§6.6.5 — RTP Hub Query

Used by GET /teams/rtp. Uses the rtp_regions_pending integer column — no JSONB path scan required.

SELECT ts.user_id, ts.rtp_regions_pending
FROM tbscan.thermal_scans ts
WHERE ts.user_id IN (
  SELECT user_id FROM tbscan.org_members WHERE org_id = :org_id
)
AND ts.is_current = true
AND ts.rtp_regions_pending > 0;
NURAI Change · §7.1 Flow A — Module 1

Module 1 — severity_level Output Added

NURAI Module 1 (Computer Vision & Thermal ROI) now classifies each region's bilateral delta into a named severity band, written synchronously to thermal_scans.measurements alongside the existing thermal fields.

Severity Classification Thresholds

severity_levelBilateral Delta
cleardelta < 0.3°C
mild0.3°C ≤ delta < 0.5°C
moderate0.5°C ≤ delta < 0.8°C
elevateddelta ≥ 0.8°C
ℹ️
Synchronous vs. Async write boundary

Module 1 writes: temp_left, temp_right, delta, affected_side, severity_level. The fields baseline_mean_delta, baseline_sd, qualifying_scan_count, and rtp_clear are written later by the async Baseline + RTP Thread (§7.4) — not by Module 1.

New Section · §7.4 Async Baseline + RTP Thread

§7.4 Async Baseline + RTP Thread — New New

Triggered by the Backend immediately after Module 1 completes and the scan row is written to DB. Runs as an asynchronous background thread concurrent with Modules 2 and 3 — does not block scan result delivery to the athlete or operator. Reads the last N measurements directly from DB; no in-memory inputs from Module 1.

1 — Read Last N Scans from DB
Queries the last N valid thermal_scans rows for the athlete from DB, ordered by generated_at DESC. The current scan row is already committed before this thread starts, so a single query covers both the new scan and its history. Builds a per-region delta history array from the result set.
2 — Compute Baselines Per Region
Rolling baseline_mean_delta, baseline_sd, qualifying_scan_count from history. These replace the retired thermal_baselines table. At ≥ 6 qualifying scans, the individual SD formula activates: T = min(100, TAI_SD / 3.0 × 100).
3 — Compute RTP Clearance Per Region
Sets rtp_clear using the 4-case logic (N/A · non-affected auto-YES · affected-side 2-scan mild/clear YES · NO). See §S1 of this document for full logic.
4 — Write Back + Alert
Patches measurements JSONB with baseline + RTP fields per region · sets rtp_regions_pending (count where rtp_clear = "NO") · sets baseline_computed_at = NOW(). If rtp_regions_pending changed vs. the previous scan, fires an rtp_clearance alert via the Alert Generator (§10).
ℹ️
T Score Formula Interaction

Module 2 during Flow A uses the baseline from the previous scan's measurements JSONB (already computed). The new scan's baseline becomes available for future T score computations. The inference path is never blocked waiting for baseline computation.

Modified Section · §7.3 Baseline Update Module

§7.3 Individual SD Baseline — Promoted to Production

The "Not Yet Implemented — Forward Spec" orange alert box is replaced. The individual SD baseline is now the active production computation.

✅
Production Status — BETA

The individual SD baseline (T = min(100, TAI_SD / 3.0 × 100)) is now active for athletes with ≥ 6 qualifying scans. The population threshold formula (T = min(100, TAI°C / 1.0 × 100)) applies when qualifying_scan_count < 6. Both values are derived from thermal_scans.measurements history — the separate thermal_baselines table is retired. Baseline computation runs in the async Baseline + RTP Thread (§7.4) after each new scan.

New Section · §10 Alert Generation System

§10 Alert Generation System — New Section New

New section inserted after §09 Notifications. Documents the background module that detects conditions requiring coaching staff attention and writes rows to tbscan.alerts.

10.1 Alert Detection Logic

Alert TypeSeverityTriggerDetection Logic
zone_change warning (→deceptive)
critical (→danger)		 Post Flow A Compare today's vs. yesterday's compass_zone in daily_assessments. If different, insert alert with metadata: {"from_zone":…,"to_zone":…}.

POST /internal/cron/alert-generate — IAM service account only. Called by Cloud Scheduler for the daily cron, and invoked inline by the Backend after Flow A and after the async Baseline + RTP thread for event-driven alerts.
