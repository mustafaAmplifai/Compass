Architecture, component interactions, data lifecycle, authentication design, and key engineering decisions for the NUR platform at the BETA release milestone.

📅 April 2026 · BETA — NUR Teams included
Section 01

Overview

NUR is a performance intelligence platform that converts rest-state thermal signals and physiological data into actionable readiness insights and customised corrective guidance for individuals and teams.

The platform captures full-body thermographic images through a guided sensor app, applies AI-driven multi-layer analysis to extract asymmetry metrics, and presents personalised readiness scores through consumer mobile and web interfaces.

1.1 Platform Goals

  • 1

    Guided Thermal Acquisition

    Deliver a structured, operator-led scan session via NUR Sensor — guiding image capture on dedicated hardware (smart mirror or tablet) and uploading thermal data securely to cloud storage.

  • 2

    COMPASS Scoring

    Compute thermal asymmetry metrics and COMPASS readiness scores via the NURAI inference service, combining thermographic signals with physiological recovery data into a single 0–100 zone classification.

  • 3

    Wearables Integration

    Ingest recovery metrics from major wearable platforms to supply the systemic readiness axis of COMPASS, with a pull-queue architecture that degrades gracefully when wearable data is unavailable.

  • 4

    Individual Readiness Insights

    Surface personalised readiness scores, scan history, trend visualisations, and AI coach recommendations to athletes through the My NUR mobile app.

  • 5

    Team-Level Intelligence

    Surface squad-level readiness analytics, thermal RTP monitoring, alert management, and clinical note authoring to coaching and clinical staff through the NUR Teams web portal.

Section 04

Authentication Architecture

NUR uses two distinct authentication strategies — OTP-driven JWT flows for user-facing clients, and a static API key for the managed NUR Sensor device.

🔑

My NUR (Consumers)

OTP → JWT access token + refresh token. Short-lived access (15–60 min), long-lived refresh. Bearer header on every request.

🔒

NUR Sensor (Device)

Static API key provisioned at device setup via MDM. Validated server-side against hashed value in GCP Secret Manager. No token refresh cycle.

👥

NUR Teams (Coaches)

OTP via SMS/WhatsApp → JWT. Same backend auth stack. GA-scoped.

📷

QR Session Flow

My NUR scans QR displayed on NUR Sensor. JWT + QR token submitted to backend. Authenticated user profile pushed to device via Firebase RTDB — no polling.

Admin Mode — API Key + Coach Identity Verification

All NUR Sensor communication uses the static API key — no JWT is issued. The OTP step verifies the coach's identity: the coach enters their email or username in NUR Sensor, receives an OTP by SMS, and submits it; the backend resolves and records the coach as operator_id on the session. Section 06

Data Architecture & NUR Metrics

This section documents the NUR data schema — redesigned around a three-layer model. Layer 1: Hardware Events (sessions) records the physical scan event and device context. Layer 2: Thermal Computation (thermal_scans) stores NURAI inference outputs from a thermogram. Layer 3: Daily COMPASS History (daily_assessments) is one authoritative computation record per athlete per calendar date — tying thermal and wearable inputs to the COMPASS score, zone, and AI coach output. Metric pipeline processing is owned and executed by NURAI; see §7.3 NURAI Metric Pipeline Module.

tbscan.thermal_scans — Layer 2: AI inference output from a single thermogram. One row per session. Referenced by multiple daily_assessments rows across up to 7 days (thermal scan reuse).

ColumnTypeDescription
Identity
thermal_scan_idUUID (PK)Primary key. Generated by NURAI when inference completes.
session_idFK → sessions, UNIQUEThe hardware session that produced this thermogram. UNIQUE enforces one thermal_scan per session. Image paths are read from sessions.gcs_path via this FK — not duplicated here.
user_idFK → auth_userDenormalized from sessions for efficient gate queries (Flow B freshness check).
Thermal Inference Outputs
t_scorenumeric(5,2), nullableNormalised Thermal Asymmetry Score (0–100). NULL if inference failed.
measurementsJSONB, nullablePer-region thermal measurements. Each key is a granular_region_id string.

Synchronous fields (written by NURAI Module 1): temp_left, temp_right, delta, affected_side, severity_level.

Async fields (written by Baseline + RTP thread): baseline_mean_delta, baseline_sd, eligible_scan_count, baseline_mode, rtp_clear.

eligible_scan_count — number of qualifying historical scans used to compute this region's baseline. Each region accumulates its own count independently; a region with ≥ 6 eligible scans uses individual SD mode automatically while regions below that threshold remain on population fallback — no global switch, no manual intervention.

baseline_mode (per-region) — individual_sd when eligible_scan_count ≥ 6 for this region; population_fallback otherwise. Stored inline so Module 2 reads the mode directly from the JSONB without re-evaluating the count at inference time.

severity_level thresholds (bilateral delta °C): clear < 0.3 · mild 0.3–0.5 · moderate 0.5–0.8 · elevated ≥ 0.8

rtp_clear values: "N/A" (eligible_scan_count < 2) · "YES" (region side ≠ affected_side — healthy contralateral, auto-clears) · "YES" (region side = affected_side AND severity_level was mild or clear for last 2 consecutive scans) · "NO" (affected side with ≥1 moderate/elevated reading in last 2 scans)

Example: {"12": {"temp_left": 36.2, "temp_right": 35.8, "delta": 0.4, "affected_side": "left", "severity_level": "mild", "baseline_mean_delta": 0.31, "baseline_sd": 0.08, "eligible_scan_count": 8, "baseline_mode": "individual_sd", "rtp_clear": "NO"}}

thermal_scans.measurements — JSONB schema

One key per detected anatomical region, keyed by granular_region_id (granular_anatomical_regions.granular_id). Replaces both asymmetry_scores (delta-only) and the eliminated thermal_measurements table (raw temps). All per-scan measurement reads are a single row fetch with no join.

    "temp_left":     36.2,          // °C — raw left-side surface temperature
    "temp_right":    35.8,          // °C — raw right-side surface temperature
    "delta":         0.4,           // |temp_left - temp_right|
    "affected_side": "left"         // "left" | "right" | "none"
  },
  "7": {
    "temp_left":     35.1,
    "temp_right":    35.3,
    "delta":         0.2,
    "affected_side": "none"
  },
  "15": {
    "temp_left":     34.9,
    "temp_right":    34.7,
    "delta":         0.2,
    "affected_side": "none"
  }

}

The async Baseline + RTP Thread (§7.4) patches this JSONB after each Flow A scan with baseline_mean_delta, baseline_sd, eligible_scan_count, baseline_mode, and rtp_clear per region — each region computed independently from its own eligible scan history. The affected_side field is computed by NURAI during inference (hotter side = affected).

6.6 Named Query Patterns

Derived read patterns used by NUR Teams API endpoints. No dedicated tables — all derived from existing schema.

§6.6.1 — 30-Day COMPASS History

Used by GET /teams/athletes/{id}/compass/history. Index reuse: idx_ud_user_date (user_id, date DESC).

SELECT date, compass_score, compass_zone, trigger_type
FROM tbscan.daily_assessments
WHERE user_id = :athlete_id
ORDER BY date DESC LIMIT 30;

§6.6.2 — Contralateral Pair Deltas

Used by GET /teams/athletes/{id}/thermal/contralateral. Returns last 14 valid scans. Pairing: regions sharing a base label (e.g. "Thigh") are grouped as left/right contralateral pairs. bilateral_diff = |left_delta − right_delta|. Response shape: { pairs: [{ label, dates[], left_deltas[], right_deltas[], bilateral_diffs[] }] }.

SELECT ts.generated_at,
       ts.measurements->:left_region_id->>'delta'  AS left_delta,
       ts.measurements->:right_region_id->>'delta' AS right_delta
FROM tbscan.thermal_scans ts
WHERE ts.user_id = :athlete_id AND ts.is_valid = true
ORDER BY ts.generated_at DESC LIMIT 14;

§6.6.3 — Scan Overdue Detection

Used by the Alert Generator cron. Compares each athlete's last valid scan date against org_settings.scan_interval_days.

SELECT om.user_id FROM tbscan.org_members om
LEFT JOIN (
  SELECT user_id, MAX(generated_at) AS last_scan
  FROM tbscan.thermal_scans WHERE is_valid = true GROUP BY user_id
) ts ON ts.user_id = om.user_id
JOIN tbscan.org_settings os ON os.org_id = om.org_id
WHERE om.org_id = :org_id
  AND (ts.last_scan IS NULL
       OR ts.last_scan < NOW() - INTERVAL '1 day' * os.scan_interval_days);

§6.6.4 — Wearable Gap Detection

Used by the Alert Generator cron. Flags active wearable connections that have not synced in over 48 hours.

SELECT wc.user_id FROM tbscan.wearable_connections wc
JOIN tbscan.org_members om ON om.user_id = wc.user_id
WHERE om.org_id = :org_id
  AND wc.status = 'active'
  AND wc.last_synced_at < NOW() - INTERVAL '48 hours';

§6.6.5 — RTP Hub Query

Used by GET /teams/rtp. Uses the rtp_regions_pending integer column — no JSONB path scan required.

SELECT ts.user_id, ts.rtp_regions_pending
FROM tbscan.thermal_scans ts
WHERE ts.user_id IN (
  SELECT user_id FROM tbscan.org_members WHERE org_id = :org_id
)
AND ts.is_current = true
AND ts.rtp_regions_pending > 0;

§6.6.6 — RTP Workflow Fetch

Used by GET /teams/rtp/{athleteId}/workflow. Returns the active workflow with inline injury data and all intervention records in a single response. Interventions are ordered chronologically.

SELECT
  w.workflow_id, w.initiated_at, w.initiated_by,
  w.injuries,
  w.cleared_at, w.cleared_by, w.manual_clear,
  json_agg(jsonb_build_object(
    'intervention_id',   iv.intervention_id,
    'intervention_type', iv.intervention_type,
    'session_date',      iv.session_date,
    'provider',          iv.provider,
    'duration_mins',     iv.duration_mins,
    'notes',             iv.notes
  ) ORDER BY iv.session_date) FILTER (WHERE iv.intervention_id IS NOT NULL) AS interventions
FROM tbscan.rtp_workflows w
LEFT JOIN tbscan.rtp_interventions iv ON iv.workflow_id = w.workflow_id
WHERE w.athlete_id = :athlete_id AND w.is_active = true
GROUP BY w.workflow_id;
Section 07

AI / ML Pipeline & NURAI

NURAI is the AI inference service deployed on Cloud Run. It is composed of three internal modules — Computer Vision & Thermal ROI, Metrics Pipeline, and GenAI Reasoning — exposed via two endpoints: POST /infer (Flow A) and POST /infer/refresh (Flow B), called exclusively by the Backend, never by clients directly. It reads all inputs from the database and GCS; it does not call any upstream wearable or Spike API.

⚠
Synchronous Processing (Current)

The pipeline is currently synchronous: the Backend waits for the NURAI inference response before returning the result preview to NUR Sensor. See Section 11, Open Question #2 for the sync vs. async decision.

7.1 Flow A — Full Pipeline triggered by NUR Sensor

Runs when an athlete completes a physical scan at a smart mirror. Executes all three NURAI modules in sequence: Computer Vision & Thermal ROI → Metrics Pipeline → GenAI Reasoning. NUR Sensor uploads the thermogram directly to GCS via signed URL — the binary never transits the Backend. The Backend then calls POST /infer { session_id } (API Key auth) to trigger NURAI. Once Module 1 completes and measurements are written to DB, the Backend immediately spawns the async Baseline + RTP Thread (§7.4) — passing the current scan's measurements as a direct input parameter alongside a zone_future (an unresolved Future/Promise). Module 2 resolves zone_future with the computed compass_zone as soon as Zone Classification finishes. The async thread's Steps 1–3 (baseline + RTP per-region) run freely without waiting; Step 4 (write-back + alert) awaits zone_future before the eligibility check, ensuring it always uses the current scan's zone — never the previous day's stale value from DB.

Module 1 — Computer Vision & Thermal ROI
ROI Extraction & Measurements
NURAI fetches the thermogram from sessions.gcs_path (GCS). Runs region-of-interest extraction to compute per-region surface temperatures and left-right deltas. Results written to thermal_scans.measurements (JSONB) along with t_score, baseline_mode, and has_skin_mask. Outputs severity_level per region based on bilateral delta: clear (<0.3°C) · mild (0.3–0.5°C) · moderate (0.5–0.8°C) · elevated (≥0.8°C). These synchronous fields are written to thermal_scans.measurements immediately. Once written, the Backend spawns the async Baseline + RTP Thread (§7.4) with two inputs: the current scan's measurements (direct in-memory, no DB re-read) and an unresolved zone_future. Modules 2 and 3 continue synchronously. Module 2 resolves zone_future with compass_zone immediately after Zone Classification — the async thread's Steps 1–3 run freely in parallel; Step 4 awaits this future before the RTP eligibility check. Baseline and RTP fields are written back to thermal_scans.measurements by the async thread upon completion.
Wearable Data Read
Reads the latest row from tbscan.daily_metrics (written by the Backend daily Spike pull — see §08). If the gap exceeds 2 days, metric_record_id is set to NULL and COMPASS scoring is skipped — thermal data is stored but no zone or score is produced.
Module 2 — Metrics Pipeline
Zone Classification (2 × 2 Matrix)
Deceptive Readiness
R ≥ 50, T ≥ 50
Danger
R < 50, T ≥ 50
Safe
R ≥ 50, T < 50
Recovery
R < 50, T < 50

After zone is determined, Module 2 immediately resolves the zone_future passed to the async Baseline + RTP Thread — unblocking that thread's Step 4 eligibility check with the current scan's zone.

Score Fusion Formula
Safe / Recovery (equal weight):
Score = floor + 24 × (0.5×R_dist + 0.5×T_dist)

Deceptive / Danger (thermal-weighted):
Score = floor + 24 × (0.3×R_dist + 0.7×T_dist)

Zone floors: Safe 76, Recovery 51, Deceptive 26, Danger 1. When thermal asymmetry is elevated, the score is pulled down even when R is high — an athlete in Deceptive Readiness cannot score above the midpoint of that band.
Module 3 — GenAI Reasoning
GenAI Reasoning Layer
Receives the computed zone, score, thermal_scans.measurements, and health context from user_profile_history as read-only inputs. Cannot alter scores or zone assignments. Produces: zone rationale, two-phase intervention plan, reassessment triggers, and coach narrative. All output written to daily_assessments.ai_coach_response (JSONB — see §06 schema).
4 — Storage & Delivery
Database Writes + FCM Push
Writes tbscan.thermal_scans (t_score, measurements JSONB, baseline_mode, generated_at). Then upserts tbscan.daily_assessments (compass_score, compass_zone, ai_coach_response, metric_record_id, trigger_type = new_thermal_scan or admin_scan). Returns 200 OK to the Backend, which dispatches the FCM push to the athlete. The async Baseline + RTP Thread was already spawned after Module 1 completed — baseline and RTP enrichment of the scan row occurs independently in the background and does not delay result delivery.

Per-Region Population Fallback

Each body region accumulates its own baseline independently. Regions with sufficient eligible scan history use the individual SD formula; regions below the threshold use the population formula (T = min(100, TAI°C / 1.0 × 100) — COMPASS §05A). Both modes resolve to the same T = 50 zone boundary. The per-region baseline_mode field in the measurements JSONB records which formula was applied; the scan-level thermal_scans.baseline_mode summarises as individual_sd, population_fallback, or mixed. Regions transition automatically as their eligible_scan_count grows — no manual intervention or global scan counter required.

7.4 Async Baseline + RTP Thread BETA

Triggered by the Backend immediately after Module 1 completes and the current scan's measurements are written to DB — not after the full Flow A pipeline. Runs as an asynchronous background thread concurrent with Modules 2 and 3. Does not block scan result delivery to the athlete or operator. Receives the current scan's measurements as a direct in-memory input from Module 1; historical scans are fetched separately from DB.

1 — Receive Current Measurements + Read Historical Scans
Accepts the current scan's measurements JSONB as a direct input from Module 1 (already in memory — no DB read for the current scan). Separately queries all prior valid thermal_scans.measurements rows for the athlete from DB, ordered by generated_at ASC. Combines both into a complete per-region delta history including the current scan.
2 — Compute Baselines Per Region
For each region independently: compute rolling baseline_mean_delta, baseline_sd, and eligible_scan_count from that region's qualifying scan history (gates: §7.3). Assign per-region baseline_mode: individual_sd when the region has sufficient eligible scans, population_fallback otherwise. Regions on different modes within the same scan are handled independently — there is no global scan count threshold.
3 — Compute RTP Clearance Per Region
For each region, sets rtp_clear:
• N/A — eligible_scan_count < 2 (insufficient history)
• YES — region side ≠ affected_side (healthy contralateral — auto-clears)
• YES — region side = affected_side AND severity_level was mild or clear for the last 2 consecutive scans
• NO — affected side with ≥ 1 moderate or elevated reading in the last 2 scans
4 — Write Back + Alert
Awaits zone_future (resolved by Module 2 immediately after Zone Classification — see §7.1). This guarantees the eligibility check uses the current scan's zone, not a stale value from DB. Once the zone is available: patches measurements JSONB with baseline + RTP fields per region · sets rtp_regions_pending (count of regions where rtp_clear = "NO") · sets baseline_computed_at = NOW(). If rtp_regions_pending = 0 AND compass_zone = 'safe', fires an rtp_milestone alert with event: "eligible_for_clearance" via the Alert Generator (§10) — notifying staff the athlete is ready to be reviewed. Clearance is never applied automatically; it requires explicit staff approval via POST /teams/rtp/{id}/clear.
ℹ️
Concurrency Model

The async Baseline + RTP Thread runs concurrently with Modules 2 and 3. Steps 1–3 (baseline computation + per-region RTP) have no dependency on Module 2 and run freely. Step 4 (write-back + alert) awaits zone_future, which Module 2 resolves immediately after Zone Classification — typically within milliseconds, well before the async thread's Steps 1–3 complete. This eliminates the race condition where Step 4 might otherwise read the previous day's compass_zone from daily_assessments before Module 2 has written the new one. Module 2 also reads the baseline from the previous scan's measurements JSONB (written by the previous async thread, already settled) — the baseline write and the scoring read operate on different scan rows, so they never race.

8.4 SPIKE Authentication Flow

SPIKE authentication involves two independent contexts that operate at different layers of the NUR stack. They must not be conflated.

ℹ
Two distinct auth contexts:
SDK / Device — My NUR uses the Spike React Native SDK to request health permissions via native OS dialogs. No token exchange, no OAuth, no server round-trip. Data stays on-device until extracted.
API / Backend — The Backend exchanges an HMAC-SHA256 signature for a Spike JWT and uses it as a Bearer token on all server-side Spike Daily Statistics API calls.

8.4.1 SDK / Device Auth (My NUR ↔ OS)

  • 1

    After SDK initialisation (see §8.1 Steps 3–4), My NUR calls spikeConnection.requestHealthPermissions(...). This launches a native OS permission dialog — HealthKit on iOS, Health Connect on Android.

  • 2

    Permissions are per-metric and managed entirely by the OS. No OAuth redirect. No callback URL. No server-side session state. Spike never receives a credential — it receives only data that the user has already approved locally.

  • 3

    Health data is stored in encrypted device storage and only extracted when My NUR explicitly calls the SDK read methods. The extracted data is then available for upload to the Backend.

  • 4

    Reauthorization: If the app is uninstalled and reinstalled, the SDK connection must be reinitialised and OS permissions re-granted. The Backend detects this via wearable_connections.status = 'error' and prompts the user to reconnect.

8.4.2 API Auth — Backend → Spike (HMAC → JWT)

The Backend authenticates against the Spike REST API using a 3-step HMAC handshake. This token is used for all server-side daily statistics pulls — the app never holds or uses it directly.

  • 1

    Generate HMAC-SHA256 Signature

    Exchange Signature for JWT

8.5 Daily Pull & Webhook Queue

Rather than blindly pulling every connected user on each cron run, the Backend uses Spike webhooks to maintain a per-user pull queue. When Spike fires a record_change event for a user, that user is marked for inclusion in the next scheduled pull. The cron job drains only queued users, keeping API call volume proportional to actual data availability.

Overview

🔌

Webhook Handler

Receives push events from Spike. Validates the X-Body-Signature HMAC, routes by event type, and writes queued user IDs to spike_pull_queue. Responds HTTP 200 immediately.

🕐

Daily Cron Job

Runs once per day (e.g. 02:00 UTC). Reads all rows from spike_pull_queue, calls the Spike Daily Statistics API for each user, writes results to tbscan.daily_metrics, then clears the queue entries.

Webhook Security — Signature Verification

Every inbound Spike webhook includes an X-Body-Signature header. The Backend must verify this before processing the payload:

import hmac, hashlib

def verify_spike_webhook(raw_body: bytes, header_signature: str, secret_key: str) -> bool:
    computed = hmac.new(
        key=secret_key.encode(),
        msg=raw_body,
        digestmod=hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(computed, header_signature)

# In the Django view:
if not verify_spike_webhook(request.body, request.headers["X-Body-Signature"], SPIKE_SECRET):
    return HttpResponse(status=403)

Webhook Handler Flow

  • 1

    Spike POSTs to POST /webhooks/spike (Backend endpoint, no auth header — authenticated via body signature). Payload may be a single object or an array of objects.

  • 2

    Backend verifies X-Body-Signature using HMAC-SHA256 with the shared secret from GCP Secret Manager. Returns 403 on mismatch; returns 200 immediately on success (processing is async).

  • 3

    For each event object in the payload, route by event_type:

    • record_change → upsert into spike_pull_queue (application_user_id, queued_at, data_start, data_end, metrics[], provider_slug)
    • provider_integration_created → mark connection active; enqueue pull
    • provider_integration_deleted → mark connection revoked; dequeue user

  • 4

    If Spike does not receive HTTP 200 within 30 s, it retries with exponential backoff: 5 s → 2 min → 30 min → 2 h → every 12 h, up to 10 attempts total. After 10 failures, the event is discarded. The Backend must be idempotent — duplicate deliveries must not create duplicate tbscan.daily_metrics rows (upsert on user_id + source_platform + metric_date + metric_type).

Daily Cron Job Flow

  • 1

    GCP Cloud Scheduler triggers POST /internal/cron/spike-pull once per day (default 02:00 UTC). The endpoint is internal-only, authenticated via GCP IAM service account.

  • 2

    Backend reads all rows from spike_pull_queue where status = 'pending'. Users with no pending queue entry (no record_change since last pull) are skipped entirely.

  • 3

    For each queued user, Backend calls the Spike Daily Statistics API using the stored JWT (wearable_connections.encrypted_access_token), scoping the request to the data_start / data_end window from the webhook payload where available.

  • 4

    Returned metrics are upserted into tbscan.daily_metrics (metric_type = 'raw', source_platform = 'spike'). activity_load (ACWR) is computed from the 7-day / 28-day history and written alongside the raw fields.

  • 5

    wearable_connections.last_synced_at is updated to NOW() on success. The processed row in spike_pull_queue is marked status = 'done' (or deleted). On API error the row is marked status = 'error' and wearable_connections.status is set to error if auth-related.

8.6 On-Demand Spike Pull

The daily cron pull (§8.5) runs at 02:00 UTC. Athletes who scan or check readiness earlier in the day may encounter a stale_wearable gate failure in Flow B if their wearable device synced data after the last cron run. The on-demand pull endpoint allows an athlete or coach to request an immediate Spike pull outside the cron schedule, making fresh wearable data available for the next POST /compass/refresh call.

Rate Limiting

To prevent Spike API exhaustion, pulls are capped at one successful pull per athlete per hour. The Backend enforces this using wearable_connections.last_synced_at. If a pull was completed within the last 60 minutes, the endpoint returns HTTP 429 Too Many Requests with a Retry-After header indicating the seconds remaining before the next pull is permitted.

Online-Only Operation

NUR Sensor requires a persistent network connection throughout the entire session. Section 10

Alert Generation System BETA

Background module that detects conditions requiring coaching staff attention and writes rows to tbscan.alerts. Runs on Cloud Scheduler (daily cron at 06:00 org timezone) and on event-driven triggers after scan completion (Flow A) and after async Baseline + RTP thread completion.

10.1 Alert Detection Logic

Alert TypeSeverityTriggerDetection Logic
zone_change warning (→deceptive)
critical (→danger)		 Post Flow A Compare today's vs. yesterday's compass_zone in daily_assessments for the athlete. If different, insert alert with metadata: {"from_zone":…, "to_zone":…}.
scan_overdue warning Daily cron Query §6.6.3: athlete's last valid scan is older than org_settings.scan_interval_days.
wearable_sync info Daily cron Query §6.6.4: wearable_connections.last_synced_at > 48 hours ago for active connections.
rtp_milestone info RTP workflow events (Backend inline) Fired on four events: (1) RTP workflow initiated (POST /teams/rtp/{id}/initiate), (2) first injury documented (POST /teams/rtp/{id}/injuries), (3) athlete thermally eligible — async Baseline + RTP Thread finds rtp_regions_pending = 0 AND compass_zone = 'safe'; fires event: "eligible_for_clearance" to prompt staff review — no clearance is applied, (4) staff approves clearance via POST /teams/rtp/{id}/clear; fires event: "cleared" with manual_clear: false (conditions met) or manual_clear: true (override).
daily_digest info Morning cron 06:00 org TZ One alert per org summarising: zone distribution, overdue scan count, wearable gap count, RTP pending count. athlete_id = NULL.

Deduplication Rule

Before inserting zone_change, scan_overdue, or wearable_sync alerts, the generator checks for an existing unacknowledged alert of the same type + athlete within the past 24 hours. If found, the new insertion is skipped to prevent alert flood.
