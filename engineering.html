NUR Platform / System Design
Document

Architecture, component interactions, data lifecycle, authentication design, and key engineering decisions for the NUR platform at the BETA release milestone.

📅 March 2026 · BETA
Section 01

Overview

NUR is a performance intelligence platform that converts rest-state thermal signals and physiological data into actionable readiness insights and customised corrective guidance for individuals and teams.

The platform captures full-body thermographic images through a guided sensor app, applies AI-driven multi-layer analysis to extract asymmetry metrics, and presents personalised readiness scores through consumer mobile and web interfaces.

1.1 Platform Goals

  • 1

    Guided Thermal Acquisition

    Deliver a structured, operator-led scan session via NUR Sensor — guiding image capture on dedicated hardware (smart mirror or tablet) and uploading thermal data securely to cloud storage.

  • 2

    COMPASS Scoring

    Compute thermal asymmetry metrics and COMPASS readiness scores via the NURAI inference service, combining thermographic signals with physiological recovery data into a single 0–100 zone classification.

  • 3

    Wearables Integration

    Ingest recovery metrics from major wearable platforms to supply the systemic readiness axis of COMPASS, with a pull-queue architecture that degrades gracefully when wearable data is unavailable.

  • 4

    Individual Readiness Insights

    Surface personalised readiness scores, scan history, trend visualisations, and AI coach recommendations to athletes through the My NUR mobile app.

Section 04

Authentication Architecture

NUR uses two distinct authentication strategies — OTP-driven JWT flows for user-facing clients, and a static API key for the managed NUR Sensor device.

🔑

My NUR (Consumers)

OTP → JWT access token + refresh token. Short-lived access (15–60 min), long-lived refresh. Bearer header on every request.

🔒

NUR Sensor (Device)

Static API key provisioned at device setup via MDM. Validated server-side against hashed value in GCP Secret Manager. No token refresh cycle.

👥

NUR Teams (Coaches)

OTP via SMS/WhatsApp → JWT. Same backend auth stack. GA-scoped.

📷

QR Session Flow

My NUR scans QR displayed on NUR Sensor. JWT + QR token submitted to backend. Authenticated user profile pushed to device via Firebase RTDB — no polling.

7.3 NURAI Metric Pipeline Module

Internal NURAI module. Accepts wearable metrics (pre-populated by the Backend's daily Spike pull) and a thermogram from GCS, executes all stages autonomously, and writes all outputs directly to tbscan.scan_results. Health questionnaire data feeds the AI Coach output only — not the R or T score.

Stage A — Data Collection
Wearable Data — tbscan.metrics
HRV, resting HR, sleep score, ACWR, and pre-computed R Score written by the Backend daily Spike pull. See §08.
Session & History Data
Thermogram GCS path from tbscan.sessions; pain entries from pain_entries; chronic conditions and demographics from users.health_history.
Stage B — Input Validation & Data Availability Gate
Wearable Data Freshness

Gap ≤ 2 days → advisory_mode = full_compass, continue. Gap > 2 days → advisory_mode = thermal_only, write compass_score = null, skip to Stage G.

Stage E — Zone Classification (2 × 2 Matrix)
Deceptive Readiness
R ≥ 50, T ≥ 50
Danger
R < 50, T ≥ 50
Safe
R ≥ 50, T < 50
Recovery
R < 50, T < 50
Stage F — COMPASS Scoring Engine
Fusion Formula
Safe / Recovery zones (equal weight):
Score = floor + 24 × (0.5 × R_dist + 0.5 × T_dist)

Deceptive Readiness / Danger zones (thermal-weighted):
Score = floor + 24 × (0.3 × R_dist + 0.7 × T_dist)

Where R_dist and T_dist are normalised distances from the zone boundary (0–1). Zone floor values: Safe 76, Recovery 51, Deceptive Readiness 26, Danger 1. When thermal asymmetry is elevated, the score is deliberately pulled down even when R is high — an athlete in Deceptive Readiness cannot score in the upper half of that band.

Per-Zone Expanded Form:

Safe (76–100):
76 + 24 × avg( (R−50)/50 , (50−T)/50 )

Recovery (51–75):
51 + 24 × avg( R/50 , (50−T)/50 )

Deceptive Readiness (26–50):
26 + 24 × ( 0.3×(R−50)/50 + 0.7×(100−T)/50 )

Danger (0–25):
25 × ( 0.3×R/50 + 0.7×(100−T)/50 )
Stage G — Storage & Delivery
Database Writes + My NUR Delivery
Single write to tbscan.scan_results: r_score, t_score, asymmetry_scores, compass_score, compass_zone, ai_coach_output (JSONB — zone response, physio guidance, coach narrative). Backend dispatches FCM push on 200 OK. See §6.5 for full schema.

7.1 Thermal Inference Pipeline

See §5.1 for the full session sequence. The inference-specific steps are:

  • NUR Sensor uploads the thermogram directly to GCS via signed URL — binary never transits the API tier.
  • Backend calls POST /infer { session_id } (GCP IAM service account). NURAI reads all inputs from DB (tbscan.metrics, pain_entries, health history) and fetches the thermogram from GCS autonomously — no further API calls.
  • NURAI writes all outputs to tbscan.scan_results in a single record, returns 200 OK; Backend dispatches FCM push to My NUR.

7.2 NURAI GenAI Layer — Stage F & G BETA

Stage F — AI Reasoning Engine

NURAI's GenAI layer receives the fully computed COMPASS assessment as a read-only input. It cannot alter scores, zone assignments, or thresholds. Its sole function is to explain and recommend based on the deterministic output from Stage E (COMPASS Scorer).

7.4 NURAI — Architecture & Intelligence Layer

ℹ
NURAI Scope

NURAI's scope is to (1) read raw wearable metrics and the pre-computed R Score from tbscan.metrics — as populated by the Backend's daily Spike pull — (2) fetch the thermogram from GCS, (3) compute all composite metrics (TAI, T Score, COMPASS Score, zone), (4) maintain individual thermal baselines (SD method, nightly batch), and (5) write all scoring and coaching outputs to tbscan.scan_results. NURAI does not call any upstream wearable or Spike API.

7.5 Thermal Baseline Update Module BETA

Nightly batch that maintains per-region, per-athlete SD baselines (COMPASS §05B). Computes mean and sample SD of left-right temperature delta per body region from qualifying "healthy reference" scans. Stored in tbscan.thermal_baselines; consumed by NURAI at inference in preference to the population threshold. Corresponds to "Individual Baseline Calculator" in §7.4 Model Registry.

⚠
Not Yet Implemented — Forward Spec

This section is a forward-looking engineering specification. The tbscan.thermal_baselines table, the nightly batch job, and the at-inference lookup described below are not yet deployed. The current production pipeline uses the population threshold formula exclusively (T = min(100, TAI°C / 1.0 × 100)). This spec defines the target behaviour once the module is implemented.

Qualifying Scan Criteria

All three gates must pass. Failing sessions are silently excluded — no errors raised, existing baselines unaffected.

GateConditionSource
No pain entryZero records in tbscan.pain_entries for this session_idpain_entries.session_id
No major asymmetryAll per-region temperature deltas < 1.0°C (below COMPASS Severe tier boundary)tbscan.thermal_measurements
Session completetbscan.sessions.status = 'completed' and no avoidance_flags indicating a compromised scantbscan.sessions

The 1.0°C gate maps to the population Severe tier boundary — scans at or above it indicate clear pathological asymmetry and must not anchor the personal normal.

At-Inference Lookup

NURAI queries tbscan.thermal_baselines WHERE is_active = true for the user. Per scanned region: if an active baseline exists, apply the individual SD formula — region_TAI_SD = (current_delta − mean_delta_c) / sd_delta_c; otherwise apply the population fallback — region_TAI_SD = current_delta_c / 0.27. Then TAI = max(all region_TAI_SD) → T = min(100, TAI / 3.0 × 100). Outputs are tagged baseline_mode = 'individual_sd' or 'population_fallback'. Both scales share the same T = 50 zone boundary (≈ 1.5 SD ≈ 0.4°C).

ℹ
Population Fallback

While a user has fewer than 6 qualifying scans for a given region, the population threshold formula is used for that region: T = min(100, TAI°C / 1.0 × 100) (COMPASS §05A). All outputs are tagged baseline_mode = 'population_fallback' until sufficient qualifying scans have been processed by the nightly batch. The fallback is removed automatically once the 6th qualifying scan is processed — no manual intervention is required.

Section 08

My NUR authenticates through the Spike SDK on-device; the Backend performs a daily scheduled pull from the Spike API and writes raw wearable metrics (including the pre-computed R Score) to tbscan.metrics (raw).     On first wearables setup, My NUR calls POST /wearables/connect. The Backend generates an HMAC-SHA256 signature using the user's ID and the Spike shared secret, exchanges it for a Spike JWT, stores it encrypted in wearable_connections, and returns { spike_token, application_id } to the app. See §8.4 for the full authentication flow.

  • 3

    My NUR initialises the Spike connection using the returned credentials: 

    const connection = await Spike.createConnectionAPIv3({
  applicationId: <application_id>,
  signature:     <hmac_signature>,
  endUserId:     <nur_user_id>
});

  • 4

    My NUR calls spikeConnection.requestHealthPermissions(...), which triggers the native OS permission dialogs — HealthKit on iOS, Health Connect on Android. No OAuth redirect. No callback URL. Permissions are granted locally on-device.

  • 5

    The Backend performs a daily scheduled pull from the Spike Daily Statistics API. Spike webhooks (record_change events) are used to build a per-user pull queue so the cron job only fetches data for users whose wearable data has actually been updated. See §8.5 for the full cron and webhook design. activity_load (ACWR) is Backend-computed from Spike history, not a direct Spike field.

  • 6

    Pulled values are written to tbscan.metrics (metric_type = 'raw') with source_platform = 'spike'. Wearable data is correlated to the nearest scan session at NURAI inference time.

8.4 SPIKE Authentication Flow

SPIKE authentication involves two independent contexts that operate at different layers of the NUR stack. They must not be conflated.

ℹ
Two distinct auth contexts:
SDK / Device — My NUR uses the Spike React Native SDK to request health permissions via native OS dialogs. No token exchange, no OAuth, no server round-trip. Data stays on-device until extracted.
API / Backend — The Backend exchanges an HMAC-SHA256 signature for a Spike JWT and uses it as a Bearer token on all server-side Spike Daily Statistics API calls.

8.4.1 SDK / Device Auth (My NUR ↔ OS)

  • 1

    After SDK initialisation (see §8.1 Steps 3–4), My NUR calls spikeConnection.requestHealthPermissions(...). This launches a native OS permission dialog — HealthKit on iOS, Health Connect on Android.

  • 2

    Permissions are per-metric and managed entirely by the OS. No OAuth redirect. No callback URL. No server-side session state. Spike never receives a credential — it receives only data that the user has already approved locally.

  • 3

    Health data is stored in encrypted device storage and only extracted when My NUR explicitly calls the SDK read methods. The extracted data is then available for upload to the Backend.

  • 4

    Reauthorization: If the app is uninstalled and reinstalled, the SDK connection must be reinitialised and OS permissions re-granted. The Backend detects this via wearable_connections.status = 'error' and prompts the user to reconnect.

8.4.2 API Auth — Backend → Spike (HMAC → JWT)

The Backend authenticates against the Spike REST API using a 3-step HMAC handshake. This token is used for all server-side daily statistics pulls — the app never holds or uses it directly.

  • 1

    Generate HMAC-SHA256 Signature

    Backend computes the signature using the Spike secret_key (from GCP Secret Manager) and the NUR user ID as application_user_id:

    import hmac, hashlib
signature = hmac.new(
    key=secret_key.encode(),
    msg=application_user_id.encode(),
    digestmod=hashlib.sha256
).hexdigest()
  • 2

    Exchange Signature for JWT

    Backend POSTs to the Spike auth endpoint:

    FieldValue
    Method / URLPOST https://app-api.spikeapi.com/v3/auth/hmac
    application_idSpike app ID from admin console
    application_user_idNUR user ID (1–128 alphanumeric chars, - _ . allowed)
    signatureHMAC-SHA256 hex digest from Step 1

    Spike responds with a JWT access token. Spike auto-creates the user on first authentication — no pre-registration required.

  • 3

    Store and Use JWT

    The JWT is encrypted and written to wearable_connections.encrypted_access_token. All subsequent Spike Daily Statistics API calls include:

    Authorization: Bearer <jwt_token>

    The token is consistent across the user's lifecycle — no refresh cycle is needed. If the token ever becomes invalid (status → error), the Backend re-runs this 3-step handshake to obtain a fresh token.

8.4.3 End-to-End Sequence

Full sequence from first wearables setup through ongoing daily pulls.

⚠
Token lifecycle: The Spike JWT is consistent across the user's lifecycle — no refresh cycle is required. If the token becomes invalid, the Backend re-runs the HMAC → JWT exchange (Steps 1–3 above) to obtain a fresh token and updates wearable_connections.encrypted_access_token. The connection_status column is set to error to signal the app to prompt re-connection if needed.

8.5 Daily Pull & Webhook Queue

Rather than blindly pulling every connected user on each cron run, the Backend uses Spike webhooks to maintain a per-user pull queue. When Spike fires a record_change event for a user, that user is marked for inclusion in the next scheduled pull. The cron job drains only queued users, keeping API call volume proportional to actual data availability.

Overview

🔌

Webhook Handler

Receives push events from Spike. Validates the X-Body-Signature HMAC, routes by event type, and writes queued user IDs to spike_pull_queue. Responds HTTP 200 immediately.

🕐

Daily Cron Job

Runs once per day (e.g. 02:00 UTC). Reads all rows from spike_pull_queue, calls the Spike Daily Statistics API for each user, writes results to tbscan.metrics, then clears the queue entries.

Webhook Events & Backend Actions

Event TypeWhen Spike Fires ItBackend Action
record_change New or updated health data arrives from a provider (wearable sync) Upsert row into spike_pull_queue for application_user_id; record earliest_record_start_at / latest_record_end_at from payload for targeted pull window
provider_integration_created User connects a new wearable provider Update wearable_connections.status = 'active'; store provider_slug; enqueue initial pull
provider_integration_deleted User removes a provider integration Update wearable_connections.status = 'revoked'; remove any pending queue entry for that user

Webhook Security — Signature Verification

Every inbound Spike webhook includes an X-Body-Signature header. The Backend must verify this before processing the payload:

import hmac, hashlib

def verify_spike_webhook(raw_body: bytes, header_signature: str, secret_key: str) -> bool:
    computed = hmac.new(
        key=secret_key.encode(),
        msg=raw_body,
        digestmod=hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(computed, header_signature)

# In the Django view:
if not verify_spike_webhook(request.body, request.headers["X-Body-Signature"], SPIKE_SECRET):
    return HttpResponse(status=403)

Webhook Handler Flow

  • 1

    Spike POSTs to POST /webhooks/spike (Backend endpoint, no auth header — authenticated via body signature). Payload may be a single object or an array of objects.

  • 2

    Backend verifies X-Body-Signature using HMAC-SHA256 with the shared secret from GCP Secret Manager. Returns 403 on mismatch; returns 200 immediately on success (processing is async).

  • 3

    For each event object in the payload, route by event_type:

    • record_change → upsert into spike_pull_queue (application_user_id, queued_at, data_start, data_end, metrics[], provider_slug)
    • provider_integration_created → mark connection active; enqueue pull
    • provider_integration_deleted → mark connection revoked; dequeue user

  • 4

    If Spike does not receive HTTP 200 within 30 s, it retries with exponential backoff: 5 s → 2 min → 30 min → 2 h → every 12 h, up to 10 attempts total. After 10 failures, the event is discarded. The Backend must be idempotent — duplicate deliveries must not create duplicate tbscan.metrics rows (upsert on user_id + spike_date).

Daily Cron Job Flow

  • 1

    GCP Cloud Scheduler triggers POST /internal/cron/spike-pull once per day (default 02:00 UTC). The endpoint is internal-only, authenticated via GCP IAM service account.

  • 2

    Backend reads all rows from spike_pull_queue where status = 'pending'. Users with no pending queue entry (no record_change since last pull) are skipped entirely.

  • 3

    For each queued user, Backend calls the Spike Daily Statistics API using the stored JWT (wearable_connections.encrypted_access_token), scoping the request to the data_start / data_end window from the webhook payload where available.

  • 4

    Returned metrics are upserted into tbscan.metrics (metric_type = 'raw', source_platform = 'spike'). activity_load (ACWR) is computed from the 7-day / 28-day history and written alongside the raw fields.

  • 5

    wearable_connections.last_synced_at is updated to NOW() on success. The processed row in spike_pull_queue is marked status = 'done' (or deleted). On API error the row is marked status = 'error' and wearable_connections.status is set to error if auth-related.

Offline / Fallback Behaviour

If NUR Sensor loses network connectivity during a session, the thermogram is queued locally and upload is retried on reconnection.
