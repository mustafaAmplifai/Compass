Section E1

System Architecture

COMPASS is a two-stage pipeline. Stage 1 is fully deterministic — it produces a COMPASS Score and Zone from raw sensor inputs with no ML involvement. Stage 2 is the Reasoning Engine — a GenAI layer that consumes the Stage 1 output as a read-only object and generates natural-language explanations and intervention plans.

🔒
Stage Isolation Requirement

The Reasoning Engine (Stage 2) must receive the compass_assessment object as a read-only payload. It must not have write access to the scoring pipeline, zone thresholds, or any input signals. Implementations that allow the GenAI layer to influence Stage 1 outputs violate the framework's clinical accountability model.

Component Overview

⌚

Wearable Ingestion Service

Receives HRV, sleep quality, resting HR, and ACWR from connected wearables. Computes the Systemic Readiness Score (0–100) and writes to the athlete state store. Expected cadence: nightly batch or on-device sync.

🌡️

Thermography Analysis Service

Processes thermography scans. Outputs the Thermal Asymmetry Index (°C delta between contralateral regions) and an array of flagged anatomical regions. Writes result + scan timestamp to athlete state store.

🧮

Scoring Engine

Deterministic. Reads the latest Readiness Score and Thermal Asymmetry Index from the athlete state store, applies the COMPASS formula (Section E2), and writes the resulting score and zone classification. See Section E2 for full algorithm.

🤖

Reasoning Engine

GenAI layer. Subscribes to new compass_assessment events, assembles the context bundle (Section E3), calls the LLM, and writes the structured explanation + intervention plan to the output store. See Sections E3–E4.

📊

Clinician Dashboard API

Read API. Surfaces the latest compass_assessment + Reasoning Engine output per athlete. Accepts clinician annotation writes which are injected into the next Reasoning Engine invocation as clinician_annotations.

🗄️

Athlete State Store

Single source of truth for all per-athlete signals and outputs. Stores 30-day rolling COMPASS history, scan history, clinician annotations, and the current active intervention plan. Time-series indexed by athlete_id.

Section E2

Scoring Engine

The scoring engine takes two normalised input axes and maps them to a 0–100 COMPASS Score and a Zone enum. The mapping is fully deterministic — no randomness, no model inference, no external calls.

Input Axes

AxisSymbolRangeDirectionComposed From
Systemic Readiness R 0–100 Higher = more ready HRV (weighted 40%), Sleep quality (30%), Resting HR (20%), ACWR inverse (10%)
Thermal Asymmetry Risk T 0–100 Higher = more inflamed TAI (°C) normalised against population or individual SD baseline (Section 05B in clinical doc)

Zone Classification Logic

Zone is assigned first, then score is computed within the zone's range. The zone boundary is determined by the two-axis quadrant.

Zone =
  SAFE            if R ≥ 50 AND T < 50
  RECOVERY      if R < 50 AND T < 50
  DECEPTIVE     if R ≥ 50 AND T ≥ 50
  DANGER         if R < 50 AND T ≥ 50

Score Formula by Zone

Score is computed within the zone's numeric band. Note that in Deceptive zone, thermal risk is weighted 70% to reflect that local tissue stress is the primary clinical concern.

SAFE (76–100):  score = round(76 + 24 × mean((R−50)/50, (50−T)/50))
RECOVERY (51–75): score = round(51 + 24 × mean(R/50, (50−T)/50))
DECEPTIVE (26–50): score = round(26 + 24 × (0.3×(R−50)/50 + 0.7×(100−T)/50))
DANGER (0–25):   score = round(25 × (0.3×R/50 + 0.7×(100−T)/50))
All inputs clamped to [0, 100] before computation. Output clamped to zone band after rounding.
⚠️
Temporal Fusion — Stale Thermal Prior

When no thermography scan is available within the athlete's zone-appropriate recency window, the engine applies a decay function to the last known TAI value before normalising. See the clinical reference document (Section 06) for full decay half-life parameters per zone. Scores generated with a decayed prior must include "thermal_prior_decayed": true in the output.

Output Object

{
  "athlete_id":          "string",
  "assessment_ts":       "ISO-8601 timestamp",
  "score":               0–100,          // integer
  "zone":                "Safe" | "Recovery" | "Deceptive Readiness" | "Danger",
  "readiness_score":     0–100,          // float
  "thermal_asymmetry":   0.0–∞,          // float, °C
  "flagged_regions":     ["string"],     // anatomical region identifiers
  "thermal_prior_decayed": boolean,
  "thermal_prior_age_days": integer | null
}
Section E3

Reasoning Engine — Context Input Model

Before generating any output, the Reasoning Engine assembles a structured context bundle from the athlete state store. Inputs are tiered by impact on reasoning quality. The engine must always proceed even when lower-priority inputs are missing, but must surface every gap explicitly in the context_gaps array of its output.

Field Type Source Role in Reasoning Priority
score integer Scoring Engine output Anchors explanation to current risk level Required
zone enum Scoring Engine output Selects reasoning path and plan type Required
readiness_score float Wearable Ingestion Service Systemic fatigue decomposition — drives sub-signal narrative Required
thermal_asymmetry float (°C) Thermography Analysis Service Local tissue stress signal — drives inflammation narrative Required
flagged_regions string[] Thermography Analysis Service Anatomical specificity — enables targeted physio plan generation High Priority
pain_injury_history object Clinician / Athlete Profile Contextual modifier — amplifies Deceptive / Danger narrative weight High Priority
demographics object Athlete Profile Population-norm adjustments in narrative (age, sex, sport, position) Recommended
acwr float Training Load System Workload context — gates progression criteria and spike warnings Recommended
days_to_competition integer | null Training Calendar Urgency modifier — affects tone and progression timeline Recommended
compass_history_30d integer[] Athlete State Store Trajectory classification — declining / stable / improving / spike Recommended
thermal_baseline_status "established" | "partial" | "none" Thermography Baseline Engine Drives uncertainty annotation — individual SD vs. population threshold Optional
clinician_annotations string | null Dashboard API write Prepended to context bundle before reasoning begins — highest precedence Optional
🔀
Graceful Degradation

When Recommended or Optional fields are absent, the engine proceeds but lowers its confidence value and adds an entry to context_gaps for each missing field. Each gap entry must include: field (the missing key), impact (plain-language effect on reasoning quality), and note (data-collection action to close the gap). The engine must never silently omit a limitation.

Context Bundle Assembly — Pseudo-code

function assembleContextBundle(athleteId: string): ContextBundle {
  const store   = athleteStateStore.get(athleteId);
  const scoring = scoringEngine.getLatest(athleteId);  // Required fields

  const bundle: ContextBundle = {
    // Required — abort if missing
    score:              scoring.score,
    zone:               scoring.zone,
    readiness_score:    scoring.readiness_score,
    thermal_asymmetry:  scoring.thermal_asymmetry,

    // High Priority — include if available
    flagged_regions:       store.latestScan?.flagged_regions       ?? null,
    pain_injury_history:   store.athleteProfile?.injury_history    ?? null,

    // Recommended — include if available
    demographics:          store.athleteProfile?.demographics      ?? null,
    acwr:                  store.trainingLoad?.acwr                ?? null,
    days_to_competition:   store.calendar?.daysToNextCompetition() ?? null,
    compass_history_30d:   store.getScoreHistory(30)               ?? null,

    // Optional
    thermal_baseline_status: store.thermalBaseline?.status         ?? null,
    clinician_annotations:   store.latestClinicianNote             ?? null,

    context_gaps: []  // populated by validateBundle()
  };

  return validateBundle(bundle);
}

Clinician annotations always win. If clinician_annotations is non-null, it is injected as the first element of the LLM prompt context, before all signal data. This ensures practitioner judgement overrides any automated interpretation — it is not a hint, it is the lead.

Section E4

Reasoning Engine — Explanation Engine

For each assessment the engine constructs a structured, multi-step reasoning chain that maps directly to named signals in the context bundle. Every claim in the output explanation is traceable to a specific input field — no inference step may introduce a signal not present in the bundle.

  1. 1
    Zone Anchor Statement

    Opens with a plain-language zone declaration derived from score and zone: "Athlete is currently in the Deceptive Readiness zone (COMPASS 38/100)…" Always generated regardless of context completeness — these are Required fields and must always be present.

  2. 2
    Signal Decomposition

    The engine maps readiness_score and thermal_asymmetry back to their contributing sub-signals (HRV, sleep quality, resting HR, ACWR, flagged regions). Each sub-signal is labelled "contributing", "neutral", or "protective" relative to the zone outcome and written into explanation.signal_decomposition[].

  3. 3
    Contextual Modifier Layer

    If High Priority or Recommended fields are present (pain_injury_history, days_to_competition, acwr), the engine applies a modifier narrative amplifying or contextualising the zone reading. Absent modifiers are listed in context_gaps — the engine must not fabricate context it was not given.

  4. 4
    Trajectory Integration

    When compass_history_30d is present, the engine classifies the trajectory as "declining", "stable", "improving", or "spike" and writes it to explanation.trajectory_note. A single Danger reading is handled differently to three consecutive Danger days — urgency tone must reflect this. If the field is absent, trajectory_note is null.

  5. 5
    Clinical Narrative Synthesis

    Steps 1–4 are synthesised into a coherent paragraph written to explanation.narrative. Tone adapts by zone: factual and reassuring for Safe, instructive for Recovery, urgent for Deceptive and Danger. In clinician_mode: true requests, clinical terminology is preferred over plain language.

  6. 6
    Uncertainty & Caveat Annotation

    The explanation closes with a calibrated confidence value ("high" / "medium" / "low") and explicit caveats. If thermal_baseline_status is "partial" or "none", the engine must note that the asymmetry reading uses a population threshold rather than individual SD. Confidence must be "low" whenever any Required field was derived from a stale or decayed prior.

Clinician annotations are injected before step 1. If clinician_annotations is non-null, it is the first element of the LLM context and sets the interpretive frame for all subsequent steps. The engine must not contradict or silently override it. The annotation is echoed back in explanation.clinician_annotation_field for audit log traceability.

Section E5

Reasoning Engine — API Contract

The Reasoning Engine exposes a single synchronous endpoint. Callers pass the pre-assembled context bundle and receive a structured response. The engine is stateless — all context must be provided in the request body.

Request

POST /v1/reasoning-engine/assess

Content-Type: application/json

{
  "athlete_id":            "string",          // required
  "assessment_ts":         "ISO-8601",        // required — timestamp of scoring event
  "context":               ContextBundle,     // required — see Section E3
  "output_format":         "full" | "brief",  // optional, default "full"
  "language":              "en",              // optional, default "en"
  "clinician_mode":        boolean            // optional — enables more technical language
}

Response

{
  "athlete_id":   "string",
  "generated_at": "ISO-8601",

  "explanation": {

    "signal_decomposition": [           // one entry per sub-signal
      {
        "signal":  "hrv" | "sleep_quality" | "acwr" | "thermal_tai" | ...,

🚨
Error: Missing Required Fields

If any of score, zone, readiness_score, or thermal_asymmetry is absent from the context bundle, the engine must return HTTP 422 with error: "MISSING_REQUIRED_CONTEXT" and a missing_fields array. It must not attempt partial generation.

Section E6

Data Sources & Integration

Each input signal has a defined owner service, update cadence, and staleness policy. Engineers integrating new data sources must register them in the athlete state store schema and map them to the corresponding context bundle field.

SignalOwner ServiceUpdate CadenceStaleness ThresholdFallback
HRV / Resting HR / Sleep Wearable Ingestion Nightly (device sync) 36 h Mark readiness_score as stale; proceed with last known
ACWR Training Load System Daily (post-session) 48 h Omit from context bundle; add to context_gaps
Thermal Asymmetry Index Thermography Analysis Per scan (athlete-initiated) Zone-dependent decay (see Section E2) Apply decay function; set thermal_prior_decayed: true
Competition Calendar Training Calendar On update N/A (static until edited) Set days_to_competition: null
Injury / Pain History Clinician Dashboard API On clinician write 90 days (prompt re-review) Omit; add to context_gaps
Clinician Annotations Clinician Dashboard API On clinician write 72 h (auto-expire unless pinned) Set clinician_annotations: null
Section E7

Error Handling

COMPASS is a clinical system. Silent failures or partial outputs are not acceptable. Every error condition must produce an explicit, structured error object — never a degraded but unmarked result.

  1. 1
    Missing Required Scoring Inputs

    If either readiness_score or thermal_asymmetry cannot be computed (no wearable sync + no valid scan), the Scoring Engine must not produce a score. Return {"status": "INSUFFICIENT_DATA", "missing": ["readiness_score" | "thermal_asymmetry"]}. The Reasoning Engine must not be invoked.

  2. 2
    Stale Thermal Prior — Decayed Below Minimum

    When the thermal prior decays below the minimum meaningful threshold (configurable per zone), the scoring engine must still compute a score using the fully-decayed value, but must set thermal_prior_decayed: true and thermal_prior_age_days. The Reasoning Engine must surface this in the explanation's uncertainty annotation and set confidence: "low".

  3. 3
    Reasoning Engine LLM Failure

    If the LLM call fails (timeout, API error, content policy rejection), the Reasoning Engine must return HTTP 503 with error: "REASONING_ENGINE_UNAVAILABLE". The compass_assessment object from Stage 1 remains valid and should still be surfaced to the clinician dashboard — the AI layer failure must not suppress the deterministic score.

  4. 4
    Context Bundle Validation Failure

    Any field that fails type validation (e.g., score outside 0–100, zone not in the enum) must cause the context bundle assembly to throw before the LLM is called. Return HTTP 422 with field-level validation errors. Do not attempt to coerce or default invalid values silently.

  5. 5
    Clinician Annotation Injection Failure

    If the clinician annotation is present but exceeds the maximum token budget reserved for the annotation slot, truncate to the budget limit and set "clinician_annotation_truncated": true in the response. Never silently drop annotation content — truncation must be visible to both the engine and the dashboard consumer.

📋
Audit Logging Requirement

Every Scoring Engine and Reasoning Engine invocation must write an immutable audit log entry containing: athlete_id, assessment_ts, all input values used, the output produced, and — for the Reasoning Engine — the context_gaps array. Log entries must not be deletable by application-layer code.
