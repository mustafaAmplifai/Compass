Section 01

Security Architecture Overview

NUR is a cloud-native platform hosted entirely on Google Cloud Platform (GCP). Security is implemented as a layered, defence-in-depth model — no single point of trust. Each architectural tier enforces its own controls independently so that a failure in one layer does not compromise the whole system.

Defence-in-Depth: Four Security Layers

Each layer enforces independent controls. A breach at one layer is contained by the layers below.

Layer 1 — Perimeter
Network & Transport Security

All communication between clients and the NUR backend is encrypted over TLS/HTTPS. No unencrypted endpoints are exposed. The NUR Sensor uploads thermal images directly to Google Cloud Storage via time-limited signed URLs — image binary data never traverses the API tier, minimising attack surface.

TLS/HTTPS — all endpoints GCS signed URLs — time-limited No public GCS bucket access Cloud Run stateless ingress
▼
Layer 2 — Identity & Authentication
Authentication Architecture

Three distinct authentication strategies are assigned per client type. Consumers use short-lived JWT tokens issued after OTP verification. Devices use pre-provisioned API keys stored as hashed values in GCP Secret Manager. Internal services authenticate via Google IAM service accounts — no shared secrets.

JWT-OTP — 15–60 min TTL Hashed API keys — GCP Secret Manager IAM service accounts — internal SHA-256 QR token hashing
▼
Layer 3 — Authorisation & Isolation
Role-Based Access & Data Scoping

Every authenticated request is checked for authorisation. Athletes see only their own data. Staff see only their organisation's data via org membership enforcement. Admin-only endpoints reject member-role tokens. The NURAI AI service is never callable directly by any client — only the backend may invoke it.

RBAC — admin / member roles Org-scoped isolation NURAI — internal only Rate limiting per athlete
▼
Layer 4 — Data & Application
Data Protection & Privacy Controls

Health data is encrypted at rest in Cloud SQL. Visual images are purged immediately after analysis — they are never stored in any database or accessible after the session ends. Sensitive tokens (OTPs, QR codes, wearable JWTs) are stored only as hashes or encrypted values. All data flows are purpose-limited by design.

Cloud SQL — encrypted at rest Visual image purge — immediate Encrypted wearable tokens Hashed OTPs — never stored raw

Section 02

Authentication & Identity Management

NUR uses three distinct authentication mechanisms, each tailored to the trust model and threat profile of its client type. No passwords are used — authentication relies on OTP verification or pre-provisioned device credentials, eliminating credential-stuffing and password-reuse risks.

2.1 — JWT-OTP Authentication (My NUR & NUR Teams)

Consumer athletes (My NUR) and clinical staff (NUR Teams) authenticate via a two-step OTP flow. A one-time passcode is delivered via email, SMS, or WhatsApp. Upon verification, the backend issues a short-lived JWT access token and a long-lived refresh token.

🔒

OTP Handling

Raw OTP codes are never stored in the database. Only otp_hash (hashed value) is persisted, with a hard expires_at TTL. Verification is a hash-compare — the plaintext OTP is discarded immediately.

📄

Token Lifetime

Access tokens expire in 15–60 minutes. Refresh tokens are long-lived but stored securely in GCP Secret Manager. This limits the exposure window if a token is ever intercepted.

👤

No Passwords

The system has no password fields, no password reset flows, and no credential stores. All session establishment requires a live OTP delivery to a verified contact method — eliminating credential-stuffing attacks entirely.

2.2 — Static API Key Authentication (NUR Sensor)

The NUR Sensor (smart mirror / operator tablet) uses a static API key pre-provisioned at device setup via MDM (Mobile Device Management). This key is never refreshed during normal operation and is validated server-side against a hashed value stored in GCP Secret Manager — the plaintext key is not stored anywhere in the database.

🔒
Hashed Validation

The server stores only the hash of the API key — not the key itself. Validation is performed by hashing the incoming value and comparing it. Even if the Secret Manager entry were exposed, the raw key could not be recovered.

2.3 — QR Code Session Flow

Individual athlete scan sessions are initiated via a one-time QR code. This is the most security-sensitive flow in the platform: it links an athlete's personal health profile to a physical mirror device. The design prevents replay attacks, race conditions, and token leakage through multiple independent controls.

  My NUR App              NUR Backend             Firebase RTDB        NUR Sensor

  KEY SECURITY PROPERTIES:
  ─────────────────────────────────────────────────────────────────
  • Raw QR token is NEVER stored in DB — only SHA-256(token) stored
  • Atomic UPDATE WHERE status='pending' prevents double-claim race
  • expires_at enforced server-side — not trusting client exp alone
  • RTDB node deleted immediately — no persistent session signalling
  • Four independent server-side validation gates before session granted

The QR token design is zero-knowledge at rest: even if the database were fully dumped, no raw QR token could be extracted or replayed. SHA-256 is a one-way function — the hash cannot be reversed to produce a valid scan payload.

Section 04

Privacy by Design

Privacy is not bolted on after the fact — it is built into the data flows of the NUR platform at the architectural level. The most sensitive data the platform handles (visual images of users' bodies) is never stored. Data collection is minimised to what is strictly necessary for wellness analysis.

4.1 — Processing-Only Visual Data Model

The NUR Smart Mirror captures standard RGB (visual) images solely to perform body-contour mapping — ensuring thermal data is correctly aligned with the user's anatomy. This is a computation input, not an output to retain.

  • RGB image: not accessible to Amplifai Health staff — ever

  • Image binary: never transits the API tier (direct GCS PUT)
  • GCS bucket: IAM-bound — no public URL, no unauthenticated access
✅
Zero Stored Visual Data

At no point in the system does a visual photograph of the user exist outside of the scan session memory. This is a deliberate architectural choice — not a configuration option. It cannot be disabled.

4.2 — Consent Management

User consent is versioned and timestamped at the database level. Every time a user accepts updated terms or a new privacy policy, the acceptance is recorded with the specific version identifier and an exact timestamp.

DB Field Purpose Example Value
privacy_policy_version Tracks which version of the Privacy Statement was accepted "pp-2026-01"
terms_version Tracks which version of the Terms of Use was accepted "tou-2026-01"
consent_accepted_at Exact timestamp of most recent consent acceptance 2026-01-07T09:15:00Z

When platform documents are updated, users are prompted to re-accept. The prior acceptance record is preserved — providing a full, auditable consent timeline for each user.

4.3 — Data Minimisation Controls

  • Wearable integration is optional — the platform operates fully without it
  • Wearable data is used solely to enhance wellness insights — never for advertising or marketing profiling
  • Pain entry can be restricted per-organisation via org_settings.allow_athlete_pain_entry
  • NURAI reads only what is needed for inference — it does not call upstream wearable APIs directly
  • The source field on pain entries distinguishes athlete self-report from staff entry
  • User profile history is append-only — prior profile states are preserved, not overwritten
Section 05

Encryption & Key Management

NUR applies encryption at every boundary: data in transit is protected by TLS, data at rest is encrypted by GCP's storage services, and sensitive tokens are hashed or encrypted before storage. All cryptographic secrets are managed exclusively through GCP Secret Manager.

Data in Transit

All communication between NUR clients (My NUR app, NUR Sensor, NUR Teams web portal) and the backend API is exclusively over HTTPS/TLS. There are no plaintext HTTP endpoints. The Spike webhook integration uses HMAC-SHA256 body-signature verification in addition to TLS.

Data at Rest — Encryption Matrix

Data / Secret Storage Encryption Method Transit Protection
Health & thermal data Cloud SQL (PostgreSQL) GCP Cloud SQL encryption at rest (AES-256) TLS
Infrared thermograms Google Cloud Storage GCS server-side encryption (AES-256) — default TLS + signed URL
Wearable OAuth JWT wearable_connections.encrypted_access_token Application-level encryption before DB write TLS
OTP codes otp_codes.otp_hash One-way hash — raw OTP never stored Delivered via email/SMS/WhatsApp over secure channels
QR session tokens session_qr_tokens.qr_payload_hash SHA-256 one-way hash — raw token never stored TLS (QR payload transmitted by app)
Device API keys GCP Secret Manager Secret Manager AES-256 encryption at rest IAM-gated retrieval over TLS
HMAC webhook secret GCP Secret Manager Secret Manager AES-256 encryption at rest IAM-gated retrieval; never exposed to clients
Spike HMAC signing key GCP Secret Manager Secret Manager AES-256 encryption at rest IAM-gated; used only server-side to generate HMAC signatures

Secret Manager as single source of truth: All cryptographic secrets — API keys, HMAC keys, wearable tokens — are stored exclusively in GCP Secret Manager. Application code retrieves them at runtime via IAM-authenticated API calls. No secrets are hardcoded in source code, environment files, or container images.

Section 06

Access Control & Authorisation

Authentication establishes identity. Authorisation determines what that identity may do. NUR enforces authorisation at multiple levels: role, organisation membership, and resource ownership.

Role-Based Access Control (RBAC)

Role Client Capabilities Restrictions
Athlete My NUR (app) View own scan reports, AI coach, connect wearable, submit pain entries, initiate scan via QR Cannot view other athletes' data; cannot access NUR Teams portal
Staff — Member NUR Teams (web) View squad dashboard, athlete detail, alerts, RTP workflows within own org Cannot log into NUR Teams — admin role required. Member role is for roster management only.
Staff — Admin NUR Teams (web) All member capabilities + roster management, org settings, scan initiation, clearance decisions Scoped to own organisation; cannot view other orgs' data
Operator (Device) NUR Sensor Generate sessions, upload thermograms, initiate admin scans for org members API Key auth only; cannot read health data; cannot invoke NURAI directly
Internal Service Cloud Scheduler, NURAI Execute scheduled jobs, run inference pipelines IAM service account auth only; no JWT/API Key pathway; not reachable by clients

Organisational Data Isolation

Every staff request is validated against the org_members table. A staff member can only access athletes who belong to their organisation. RTP workflows, clinical notes, alerts, and settings are all scoped to org_id. There is no cross-organisation data visibility by design.

🏠

Org-Scoped Data Access

All staff API endpoints validate org membership before returning data. A staff member from Organisation A cannot query athletes belonging to Organisation B, regardless of their role.

⚙

Internal Endpoint Lockdown

Cron endpoints (/internal/cron/*) are protected by GCP IAM service account authentication. Cloud Scheduler presents a valid service account identity with each call. Requests without a valid IAM token are rejected before reaching application code.

📋

NURAI Isolation

The NURAI inference service is never exposed to any client. It is only invocable by the backend via GCP IAM. This ensures that AI inference cannot be triggered directly or manipulated from outside the authorised backend flow.

⌛

Rate Limiting

Wearable on-demand sync is limited to one successful pull per athlete per hour. On Spike API error, the rate-limit window is not consumed — enabling immediate retry without penalty.

Section 07

Third-Party Integration Security

NUR integrates with Spike (wearable data aggregator) for optional wellness metrics. This integration is built with multiple security layers: HMAC webhook authentication, encrypted token storage, idempotent data writes, and graceful degradation when the third-party service is unavailable.

Spike Webhook Verification

All inbound webhooks from Spike are authenticated via HMAC-SHA256 body signature. The shared secret is stored in GCP Secret Manager and never exposed outside the backend service. No webhook event is processed without a valid signature.

       |                               |     from GCP Secret Manager  |
       |                               |                              |
       |                               |  2. Compute HMAC-SHA256      |
       |                               |     of raw request body      |
       |                               |                              |

  ─────────────────────────────────────────────────────────────────
  5s → 2min → 30min → 2h → every 12h — max 10 attempts

Wearable OAuth Token Security

  • Spike JWT stored as encrypted_access_token — application-level encryption before DB write
  • Token decrypted only at the point of a Spike API call — never cached in memory beyond the request
  • Connection status tracked: pending → connected → active → revoked → expired
  • On provider_integration_deleted webhook: connection marked revoked, user dequeued from sync
  • HMAC signature regenerated if token becomes invalid — no stale credential persists

No Data Resale Commitment

✅
Wearable data is never sold or used for advertising

Per the NUR Privacy Statement (effective January 7, 2026): wearable data collected through integrations is used solely to enhance wellness insights, correlate physiological signals with thermal analysis results, and generate wellness indicators within the NUR platform. It is not used for advertising, marketing profiling, or resale.

Graceful Degradation

If wearable data is unavailable or has a gap greater than 2 days, NURAI automatically falls back to thermal-only COMPASS scoring. There is no error state — the platform continues to deliver value with reduced inputs. A NULL in metric_record_id signals a thermal-only computation without requiring a separate mode flag.

Section 08

Audit Trail & Immutability

NUR maintains a comprehensive, append-only audit record across its most sensitive operations: AI inference, clinical documentation, wearable data sync, return-to-play workflows, and user consent. Immutability is enforced at the data model level — not as a policy that can be bypassed.

AI Algorithm Versioning

Every thermal scan record stores the exact version of each AI/ML component that produced it. This enables full reproducibility audits and version-based investigation if algorithm behaviour needs to be reviewed retrospectively.

Field Table Purpose
segmentation_algorithm_version thermal_scans Computer vision pipeline version used for body-region extraction
metric_fusion_algorithm_version thermal_scans COMPASS score fusion formula version (thermal + wearable weighting)
reasoning_model_version thermal_scans GenAI model version used for AI coach narrative generation
baseline_computed_at thermal_scans Timestamp when async baseline computation thread completed
generated_at daily_assessments.ai_coach_response (JSONB) Exact time of GenAI response generation — embedded in the response itself

Clinical Notes — Append-Only Immutability

Clinical notes authored by staff are stored with a strict append-only constraint at the database level. No UPDATE operation is permitted on the clinical_notes table. Every note permanently records its author, timestamp, and note type. This ensures that clinical documentation cannot be altered retroactively.

Return-to-Play Workflow Audit

The RTP workflow captures a full chain of custody for injury management decisions:

  • initiated_by — staff member who opened the workflow
  • initiated_at — exact timestamp of workflow creation
  • cleared_by — staff member who authorised return to play
  • cleared_at — exact timestamp of clearance
  • manual_clear — boolean flag distinguishing manual override from algorithm-guided clearance
  • Full intervention log in rtp_interventions — session date, provider, duration, notes, recorded_by

Wearable Sync Audit Log

Every Spike data pull — whether scheduled, webhook-triggered, or on-demand — is recorded in the sync_logs table. This provides a complete history of all wearable data ingestion events.

Field Purpose
sync_type scheduled | webhook_triggered | on_demand
status success | failed | partial
records_synced Count of metric records written in this pull
started_at / completed_at Exact pull duration timestamps
error_message Failure reason when status = failed or partial
spike_response (JSONB) Full raw Spike API response — preserved for debugging and compliance

Profile History Versioning

The user_profile_history table is append-only: when a user's profile changes (pain entries, activity level, health history), a new row is written and is_current is toggled. Prior versions are preserved permanently. Each daily_assessment records a profile_id FK — capturing exactly which profile version was active at the time of computation, enabling full reproducibility of historical COMPASS scores.

Section 09

Operational Security

Security extends beyond software controls into operational procedures: how devices operate, how sessions are managed, and how the system protects against real-world misuse scenarios.

📷

Online-Only Device Operation

The NUR Sensor has no offline fallback and performs no local image queuing. If network connectivity is lost, scanning stops. This eliminates the risk of sensitive health data accumulating on a physical device outside of a secure network context.

🛑

Single-Use QR Tokens

Session QR codes are single-use — they are atomically marked as claimed on first use. A second claim attempt returns HTTP 409. Expired tokens return HTTP 410. This prevents replay attacks even if a QR code is photographed or intercepted.

🔥

Firebase RTDB Ephemeral Nodes

The RTDB node used to signal session readiness to the NUR Sensor is deleted by the backend immediately after the athlete's profile has been delivered. RTDB security rules enforce TTL independently. No persistent session state remains in Firebase after scan initiation.

🏗

MDM Device Provisioning

NUR Sensor API keys are pre-provisioned at device setup through MDM (Mobile Device Management). This ensures keys are managed through an enterprise device management system rather than entered manually, reducing the risk of key exposure during provisioning.

🔔

Alert Deduplication

The alert system suppresses duplicate alerts of the same type for the same athlete within a 24-hour window. This prevents notification fatigue without suppressing genuinely new clinical events.

🚩

Avoidance Flags

Sessions carry avoidance_flags to signal known scan quality issues (e.g. inadequate acclimatisation, environmental interference). Flagged sessions can be retrospectively invalidated via thermal_scans.is_valid = false without deletion, preserving the audit record.

Scan Environment Controls

The Terms of Use impose procedural controls that complement technical security measures:

  • Scans must be conducted only in designated private, enclosed rooms — reducing ambient thermal interference and protecting user privacy
  • Mandatory acclimatisation period enforced by the app before scanning begins — protecting data accuracy
  • Users are responsible for not sharing their Session ID or account credentials — preventing session hijacking

10.2 — Privacy by Design (Cavoukian's 7 Foundational Principles)

Principle 1

Proactive, Not Reactive

Visual image purging is an architectural decision, not a reactive deletion policy. It is impossible for images to accumulate — they are never written to persistent storage in the first place.

Principle 2

Privacy as the Default Setting

Wearable integration is opt-in. Athlete pain entry can be disabled per-org by default. No data collection occurs beyond what is required for the session. Users must actively consent before any health profile is linked to a device.

Principle 3

Privacy Embedded into Design

The data model enforces privacy structurally: OTPs stored as hashes, QR tokens stored as hashes, clinical notes append-only, profile history versioned. Privacy cannot be disabled by configuration.

Principle 4

Full Functionality — Positive-Sum

Privacy controls do not degrade platform capability. Thermal-only scoring functions when wearable data is absent. The platform delivers full wellness insights with zero retained visual imagery.

Principle 5

End-to-End Lifecycle Security

Data is protected from collection (TLS, signed URLs) through processing (in-memory image purge, IAM-bound GCS) to storage (encrypted DB) through audit (append-only logs) and eventual deletion (status lifecycle tracking).

Principle 6

Visibility & Transparency

Consent is versioned and timestamped. Users are informed at point of use exactly what data is collected and for what purpose. The Privacy Statement explicitly documents the image purge policy, data sharing scope, and wearable data usage limits.

Principle 7

Respect for User Privacy

Personal health data is never sold to advertisers. Wearable data is used solely for wellness insights. Users can control pain entry permissions. Consent can be re-granted per updated version.
